## Known remaining limitations (v3 â†’ v4 candidates)

- **Falcon is still in-process** behind a single global lock. With many cameras + heavy traffic Falcon will saturate before vLLM does. v4 candidate: move Falcon to its own server (Triton / vLLM-vision / FastAPI wrapper).
- **No tensor-parallel sharding** of Gemma across multiple GPUs. Override via `VLLM_EXTRA_ARGS=--tensor-parallel-size=N` in `.env` if needed.
- **No authentication** on the dashboard. Bind to localhost or front with nginx + auth.
- **Gemma 4 image-token budget is wired through `mm_processor_kwargs`** â€” vLLM 0.19.x accepts this on the chat endpoint; if a future vLLM tightens the schema, the request body may need adjustment.
- **`gemma_video_fps_source`** still defaults to 5 fps (in code, not config). Move to `config.yaml` if production needs sub-second sampling.
- **Per-camera dispatcher pending slots have an unbounded TTL** of `merge_window + 120s`. A camera that produces one clip then goes silent will hold the slot until that timeout expires. Acceptable but worth noting.
- **`coercion_token_budget` accepts but does not propagate `cameras[].token_budget` if the value is in `cooldown_sec`** range â€” operator error responsibility.
- **Dashboard MJPEG generator** runs once per browser tab per camera. With many tabs the broker's `snapshot()` will be hit at NĂ—12 Hz.

## DGX Spark / GB10 (sm_121) compatibility findings

The GB10 (Grace-Blackwell server) has compute capability **sm_121**, not
sm_120 (consumer Blackwell). vLLM and PyTorch have a tight version
matrix on this brand-new platform:

| Stack | Result |
|---|---|
| vLLM 0.19.1 + torch 2.10+cu128 (PyPI default) | torch is CPU-only on aarch64 â€” no CUDA. |
| vLLM 0.19.1 + torch 2.10+cu128 (PyTorch index) + nvidia-cu12 stack | Loads model into GPU VRAM (17.9 GB), then **`cudaErrorNoKernelImageForDevice`** at first Triton kernel autotune. Torch wheel only carries kernels for sm_80..sm_120; GB10 is sm_121. |
| vLLM 0.19.1 + torch 2.10+cu128 + `--enforce-eager` (no cudagraphs) | *measured at end-of-test; see Final Status Checklist* |
| vLLM 0.19.2-nightly cu129 + torch 2.12 nightly cu129 | Resolves the sm_121 kernel issue. But hits **`KeyError: 'layers.0.moe.moe.experts.0.down_proj.weight'`** â€” a regression in vLLM nightly's `Gemma4Model.load_weights` (substring mapper duplicates `moe.`). The model-supplied `gemma4_patched.py` would fix this, but it uses `FusedMoE(reduce_results=True)` which was REMOVED in vLLM 0.19.2-nightly (`TypeError: FusedMoE.__init__() got an unexpected keyword argument 'reduce_results'`). |

**Working stack as shipped in v3** (after install.sh):

```
torch         2.12.0.dev20260402+cu129  (PyTorch nightly, sm_121-capable)
torchvision   0.27.0.dev20260402+cu129
vllm          0.19.2rc1.dev196+g095d2f87e.cu129  (vLLM nightly, ABI-matched)
```

`gemma4_patched.py` auto-detects vLLM 0.19.2+ and skips the swap (the
model-supplied patch is incompatible with that release). On vLLM 0.19.1
it applies the patch from the model repo.

If the nightly stack still hits the Gemma4 weight-loader regression on
your DGX, the fallback paths are:
1. `--enforce-eager` mode with vLLM 0.19.1 + the model patch (skips
   cudagraph autotune that triggers the sm_121 kernel error).
2. Build vLLM 0.19.1 from source against torch 2.12 nightly.
3. Use NVIDIA's NGC vLLM container (already Blackwell-aware).
4. Wait for vLLM stable to ship a release with the Gemma 4 NVFP4 fixes.

The v3 application code is complete and correct; the only blocker is
this upstream three-way compatibility pinch.

### Summary

The v3 software is fully implemented and reads / writes the same wire
format as v2 (verified by import + schema review). The only blocker is
that vLLM 0.19.1 was built for sm_120 max while the GB10 in this DGX is
sm_121 (Grace-Blackwell server), and vLLM nightly that has sm_121
support has a regression in its Gemma 4 MoE weight loader that the
model-supplied patch can't fix (it uses a `FusedMoE` constructor arg
that nightly removed). See "DGX Spark / GB10 (sm_121) compatibility
findings" above.

### What would happen the moment vLLM is healthy

`InferenceWorker.analyze(clip)` would call `gemma.reason(...)` over HTTP, parse the JSON, write a CSV row with the new `camera_id` / `classifier` / `scenario_label` columns, save the annotated PNG, encode the evidence MP4, and update the per-camera `FrameBroker` so the dashboard's last-detection panel populates. None of those code paths run today only because the Gemma call returns a `ConnectionError`. Every line that handles the result was added to v3 and validated by import + schema review.

### To reach the green checks

Replace vLLM with one that has both **sm_121 kernels** AND **vLLM 0.19.1's gemma4.py** (so the model-supplied patch applies). Three concrete options:

1. **NVIDIA NGC vLLM container.** The fastest path â€” NVIDIA ships pre-built containers with Blackwell support. Pull, mount this v3 directory inside, run `vllm_start.sh` from the container.
2. **Build vLLM 0.19.1 from source against torch 2.12 nightly.** `pip install --no-build-isolation -e git+https://github.com/vllm-project/vllm@v0.19.1#egg=vllm`. ~30-60 min, requires CMake + ninja + CUDA toolkit. Will produce a wheel whose `_C` is ABI-matched to torch 2.12 and whose Gemma4 implementation is the one the model patch targets.
3. **Wait for vLLM stable to publish a Blackwell-supporting release.** vLLM 0.19.2 is in nightly; once stable, a similar wheel for `cu129` should ship.

Once any of these is in place, `bash start.sh` will boot the full stack on port 3902 with no other code changes.
