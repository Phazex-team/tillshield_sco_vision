  the whole 58G safetensors + materialises the model. `dtype` vs
  `torch_dtype` and `device_map=auto` did NOT fix it.
- **Fix:** `reasoning/providers/qwen3_vl.py` supports `load_in_4bit`
  (BitsAndBytesConfig nf4, double-quant). `config.yaml`

  fits with the recorder paused; resident is lower. Tried pre-quant +
  save — bnb `save_pretrained` did NOT compress on this version (~56G),
  so we stayed on **on-the-fly 4-bit**.
- Reasoning chain restored to `qwen3_vl → gemma` (Gemma is a lazy HTTP
  fallback on :8001, never loaded unless Qwen errors).

- Stage A (search) and Stage B (export) both ON; local recorder remains
  the fallback and is **not** disabled.

## How to run (next time)
```

FRAUD_OFFLINE_MODE=1 FRAUD_CONFIG=./config.yaml \
  venv/bin/python scripts/run_app.py --host 127.0.0.1 --port 3902
# segment recorder for the return-counter camera
venv/bin/python scripts/run_segment_recorder.py --camera-id cam_return_01 --segment-duration-sec 60
# reviewer UI:  http://127.0.0.1:3902/review.html
# trigger analysis (incl. NVR retrieval): the "Request reprocess" button,
#   or POST /api/v1/cases/{id}/reprocess
```
Reprocess loads Qwen 4-bit (~5 min) + NVR export (~real-time for the
window), so it takes several minutes per case.
